<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local config → FTP driver | Username, password, timeout, passive-mode intent, and remote paths enter the driver from untrusted configuration |
| FTP server → driver metadata | Remote LIST/MLST/MDTM responses can be incomplete, stale, or misleading |
| local filesystem → FTP upload | Local file contents are streamed to a remote server over plain FTP |
| FTP transport → reconnect logic | Network interruptions can corrupt state or leave a stale control connection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T | `driver/ftp/ftp.go` connect/config handling | mitigate | Validate required FTP credentials and timeout format at connection setup; reject unsupported non-passive mode with explicit error instead of silently changing semantics |
| T-02-02 | I | FTP metadata handling | mitigate | Prefer `GetTime` only when supported, treat timestamps as coarse otherwise, and bias toward retransferring rather than skipping changed files |
| T-02-03 | D | Reconnect path | mitigate | Serialize connection access with a mutex, bound reconnects to the injected `retry.Retry`, and retry the failed operation at most once after reconnect |
| T-02-04 | R | Unsupported operations (`Symlink`, unsupported `Chtimes`) | mitigate | Return explicit operational errors instead of nil so logs and summaries preserve the true backend capability surface |
| T-02-05 | I | Plain FTP credentials and data in transit | accept | Plain FTP is an explicit v1 project constraint; do not add FTPS in this phase, but keep the limitation documented in research and later user docs |
</threat_model>
